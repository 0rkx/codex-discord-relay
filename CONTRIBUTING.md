# Contributing to Relay

Relay controls a local Codex runtime from Discord. Changes that look like presentation work can
alter authorization, local execution, or secret handling. Keep patches small, typed, and tested.

## Development environment

- Rust 1.88 or newer
- Windows 10/11 for installer, Desktop discovery, Credential Manager, ACL, and startup tests
- Codex Desktop for ignored live app-server tests
- A disposable private Discord server/bot for manual Discord flows

Fork the repository through its hosting UI, clone that fork, and run from the checkout:

```powershell
cargo test --all-targets --locked
```

Do not use a production Discord token in tests. Never commit `.env`, SQLite files, local config,
task content, attachment caches, or credentials.

## Change workflow

1. Open an issue for broad behavior or protocol changes. Security reports must follow
   [SECURITY.md](SECURITY.md), not a public issue.
2. Create a focused branch from the current default branch.
3. Add or update tests with the behavior change.
4. Update docs when command UX, config, permissions, data retention, protocol coverage, or security
   behavior changes.
5. Run the required checks.
6. Open a pull request describing user impact, security impact, validation, and live tests run.

## Required checks

```powershell
cargo fmt --all -- --check
cargo test --all-targets --locked
cargo clippy --all-targets --all-features --locked -- -D warnings -A clippy::pedantic
```

For protocol, runtime discovery, setup, Discord, or GOD changes, also run when available:

```powershell
cargo test --lib --locked -- --ignored --nocapture
cargo build --release --locked
.\target\release\codex-discord.exe doctor --deep
```

Pull-request results list which live checks ran and which were unavailable.

## Project invariants

- Relay is one Rust process plus the native Codex companion.
- Authorization and execution-policy decisions have one shared classification path.
- The crate forbids unsafe code.
- Stable app-server flows use typed parameter structures; compatibility flows validate against the
  installed schema before dispatch.
- Unknown methods, shapes, owners, guilds, channels, and privileged states fail closed.
- GOD passwords enter through a Discord modal or local standard input and persist only as an
  Argon2id hash.
- Durable delivery uses idempotent keys, bounded retries, and visible dead letters.
- Discord mirror pruning never deletes the underlying Codex task.

## Adding or changing a Codex method

1. Inspect schema generated by the installed app-server.
2. Add or update the audited action specification: family, label, input strategy, authorization,
   confirmation, and result renderer.
3. Add a typed helper for common stable flows; use schema-derived fields for compatibility shapes.
4. Add parameter validation and authorization tests.
5. Update server-request/notification dispositions if Codex can initiate the method.
6. Run coverage tests. Generic-only methods are reported separately and need an audited spec before
   release.
7. Describe Discord UX separately from schema-routing coverage.

## Discord UX changes

Use native embeds, components, modals, autocomplete, and attachments when they improve the flow.
Respect Discord limits and bind every component to the owner and task context. Verify:

- control-channel text cannot dispatch accidentally;
- component tokens expire and cannot cross users/tasks;
- long content remains recoverable as a complete attachment;
- deleted/pruned channels detach their task mapping;
- restart reconciliation does not duplicate a side effect;
- private overwrites remain correct.

## Pull request checklist

- [ ] Scope is focused and unrelated user work is untouched.
- [ ] Tests prove behavior and failure path.
- [ ] Formatting, tests, and clippy pass with the locked dependency graph.
- [ ] Live checks and installed Codex version are recorded, or skips are explicit.
- [ ] Security, privacy, migration, and rollback impact are described.
- [ ] Documentation/changelog are updated when public behavior changed.
- [ ] Diff and artifacts contain no token, password, private Discord ID, task data, or local path.
